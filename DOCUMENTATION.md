# Ovra Backend — полная документация

Бэкенд сервиса **Ovra**: превращает события из Telegram-чатов и встреч в задачи и
карточки в [YouGile](https://yougile.com). Это серверная часть (роль «У1») —
API-шлюз, очередь, воркер, хранилище и интеграция с YouGile в одном Go-бинаре.

## 1. Что это и на что способен

Ovra-бэкенд — **многотенантный** сервис (один Telegram-чат = один воркспейс = один
проект YouGile). Что он умеет уже сейчас:

- **Принимать события** от бота (У3) и воркера встреч (У2) через `POST /v1/events`
  и обрабатывать их асинхронно очередью + воркером.
- **Создавать задачи**: по одобрению хоста задача сохраняется в БД и публикуется
  **карточкой в YouGile** в колонку «Сделать».
- **Двигать карточки по статусам** (`todo → in_progress → review → done`): смена
  статуса задачи перемещает карточку в соответствующую колонку и закрывает её при
  `done` (нужно для вечерней сверки, FR-7).
- **Маппить исполнителя**: имя из задачи (от Claude) сопоставляется с пользователем
  YouGile и назначается на карточку.
- **Хранить креды YouGile на каждый воркспейс** в зашифрованном виде (AES-256-GCM),
  собирая их у хоста при онбординге двумя путями (готовый ключ или логин/пароль).
- **Отдавать дайджест** задач воркспейса через `GET /v1/workspaces/{tenant}/tasks`.
- **Бутстрапить YouGile** с нуля (проект + доска + 4 колонки) утилитой `cmd/ygsetup`.

Что **пока заглушки** (точки расширения для команды): обработчики событий
`chat_message` и `transcript_ready`, таблица `meetings`, вызов Claude.

## 2. Архитектура

**Модульный монолит** на Go: один бинарь, слои разделены интерфейсами. Это даёт
быструю сборку/отладку для хакатона и бесшовный переход к продакшн-архитектуре.

```
                 ┌─────────────────────────────────────────────────┐
   Telegram-бот  │                  Ovra backend                   │
   (У3) ───────► │                                                 │
                 │   ┌────────────┐   publish   ┌──────────────┐   │
   Воркер встреч │   │ API-шлюз   │ ──────────► │   Queue      │   │
   (У2) ───────► │   │ (net/http) │             │ (in-memory)  │   │
                 │   └─────┬──────┘             └──────┬───────┘   │
                 │         │                           │ subscribe  │
                 │         ▼                           ▼            │
                 │   ┌────────────┐            ┌──────────────┐    │
                 │   │  Service   │ ◄───────── │   Worker     │    │
                 │   │  (логика)  │            │ (роутинг по  │    │
                 │   └─────┬──────┘            │   type)      │    │
                 │         │                   └──────────────┘    │
                 │    ┌────┴─────┐                                  │
                 │    ▼          ▼                                  │
                 │ ┌────────┐ ┌──────────────┐                     │
                 │ │Postgres│ │YouGile client│ ───► YouGile API   │
                 │ │(pgx)   │ │(REST v2)     │                     │
                 │ └────────┘ └──────────────┘                     │
                 └─────────────────────────────────────────────────┘
```

**Два «шва» под продакшн:**

- **`Queue`** — интерфейс `Publish`/`Subscribe`/`Close`. Сейчас реализация на
  Go-каналах; замена на NATS JetStream = новая реализация интерфейса, без правки
  продюсеров/консьюмеров.
- **`Repository`** — интерфейс доступа к данным. Реализация на Postgres (pgx);
  при необходимости можно подменить (например, SQLite) без изменения логики.

## 3. Структура проекта

```
cmd/server/main.go              — точка входа: конфиг → БД → миграции → сидинг → очередь → HTTP
cmd/ygsetup/main.go             — утилита бутстрапа YouGile (проект/доска/колонки)
internal/config                 — env + workspace.yaml → Config
internal/domain                 — сущности: Workspace, User, Task, Meeting
internal/storage                — интерфейс Repository + Postgres (pgx) + раннер миграций
internal/secret                 — AES-256-GCM шифрование секретов (токены YouGile)
internal/integrations/yougile   — REST-клиент YouGile API v2
internal/columns                — сопоставление колонок доски с каноническими статусами
internal/integrations/llm       — опц. AI-классификатор колонок (OpenRouter, OpenAI-совместимый)
internal/service                — бизнес-логика: публикация и смена статуса задач
internal/queue                  — интерфейс Queue + in-memory (каналы)
internal/worker                 — потребитель очереди, роутинг по type
internal/transport/http         — API-шлюз: хендлеры, middleware, валидация
migrations/                     — SQL-миграции (вшиты в бинарь через embed)
docker-compose.yml              — app + postgres
workspace.yaml                  — каталог тенантов (id, чат, проект, колонки)
```

## 4. Как запустить

### Вариант A — Docker (одной командой)

```bash
docker compose up --build
```

Поднимет Postgres + приложение, применит миграции и засеет воркспейсы из
`workspace.yaml`. Проверка:

```bash
curl http://localhost:8080/healthz
# {"status":"ok","workspaces":1}
```

### Вариант B — локально (без Docker)

Нужен запущенный Postgres — можно поднять только сервис БД из compose:

```bash
docker compose up -d postgres     # Postgres на ХОСТОВОМ порту 5433
go run ./cmd/server
curl http://localhost:8080/healthz
```

> **Порт 5433:** хостовый порт Postgres специально не 5432, чтобы не конфликтовать
> с локально установленным PostgreSQL. Внутри compose-сети приложение всё равно
> ходит на `postgres:5432`.

### Первый запуск с нуля (демо за 5 минут)

```bash
# 1. поднять БД
docker compose up -d postgres

# 2. создать проект+доску+колонки в YouGile и получить их ID
YOUGILE_LOGIN=you@example.com YOUGILE_PASSWORD=*** go run ./cmd/ygsetup
#   → печатает yougile_project_id и ID колонок — вставить в workspace.yaml

# 3. запустить сервер (нужен APP_SECRET для хранения кредов)
APP_SECRET=change-me go run ./cmd/server

# 4. сохранить креды воркспейса
curl -X POST localhost:8080/v1/workspaces/ws-demo/credentials \
  -H 'Content-Type: application/json' \
  -d '{"login":"you@example.com","password":"***"}'

# 5. создать задачу → карточка появится в YouGile
curl -X POST localhost:8080/v1/tasks -H 'Content-Type: application/json' \
  -d '{"tenant_id":"ws-demo","title":"Первая задача","assignee":"Иван"}'
```

## 6. Подключение YouGile (онбординг)

Токен YouGile хранится **на каждый воркспейс**, а не глобально (каждая команда — свой
аккаунт YouGile). Бот собирает креды у хоста и шлёт на бэк; в БД лежит только
**API-ключ, зашифрованный AES-256-GCM**. Пароль не сохраняется никогда.

Два пути:

```bash
# 1) готовый API-ключ — работает для ЛЮБОГО аккаунта (Google/Яндекс/VK/email)
curl -X POST localhost:8080/v1/workspaces/ws-demo/credentials \
  -H 'Content-Type: application/json' -d '{"api_key":"<ключ>"}'

# 2) логин/пароль — бэк сам получит companyId и сгенерирует ключ
#    (company_name — опционально, если у аккаунта несколько компаний)
curl -X POST localhost:8080/v1/workspaces/ws-demo/credentials \
  -H 'Content-Type: application/json' \
  -d '{"login":"you@example.com","password":"***","company_name":"Acme"}'
```

> **Соцвход (Google/Яндекс/VK):** путь логин/пароль требует заданного пароля. Если
> человек входит только через соцсеть, пароля может не быть — тогда используется
> путь **`api_key`** (ключ генерируется в настройках YouGile независимо от способа
> входа). Этот путь рекомендуется как основной: он универсален и безопаснее.

### Сопоставление колонок доски

У разных команд колонки названы по-разному («Сделать» / «To Do» / «Backlog»), а Ovra
оперирует четырьмя каноническими статусами `todo / in_progress / review / done`.
Поэтому при подключении к **существующей** доске её колонки нужно один раз сопоставить
со статусами. Это делает эндпоинт `POST /v1/workspaces/{tenant}/board/resolve`.

**Как резолвится** (детерминированно, без ИИ — это разовый шаг онбординга):

1. **Словарь синонимов** (RU + EN) с нормализацией названия (lowercase, срез эмодзи/
   пунктуации): `сделать/backlog/очередь → todo`, `в работе/doing/wip → in_progress`,
   `ревью/qa/проверка → review`, `готово/done/закрыто → done`.
2. **Ординальный фолбэк** для непонятных колонок: левая → `todo`, правая → `done`,
   средние распределяются. На реальных канбанах порядок «слева направо по прогрессу».

Ответ содержит `confident` (всё легло по словарю — можно доверять без проверки),
`complete` (все 4 статуса заполнены) и `assignments[]` — как и каким методом
(`dictionary` / `ordinal`) легла каждая колонка. Маппинг сразу сохраняется в
`workspaces.col_*`.

```bash
# авто-режим: распознать первую доску проекта
curl -X POST localhost:8080/v1/workspaces/ws-demo/board/resolve
# → {"confident":true,"complete":true,"board_id":"...","mapping":{...},"assignments":[...]}

# ручной override: хост поправляет маппинг явными column id
curl -X POST localhost:8080/v1/workspaces/ws-demo/board/resolve \
  -H 'Content-Type: application/json' \
  -d '{"todo":"<col>","in_progress":"<col>","review":"<col>","done":"<col>"}'
```

> **Новые доски** проще: `cmd/ygsetup` сразу создаёт 4 канонические колонки, и резолв
> для них тривиален. Эндпоинт нужен в первую очередь для **готовых** досок.
>
> **Что должен сделать бот (У3):** после сохранения кредов дёрнуть `board/resolve`; если
> `confident:false`, показать хосту `assignments[]` на подтверждение и при правке
> отправить override.

### Реестр участников (кто кому назначается)

Есть **две независимые связи**, не путать:

1. **Воркспейс ↔ YouGile** — делает только **руководитель** (один раз отдаёт креды). Один
   токен на весь воркспейс. Рядовым участникам подключаться не нужно.
2. **Человек ↔ (Telegram + имя + YouGile-юзер)** — нужно для назначения задач и `@`-тегов.
   Бот **пассивно** регистрирует людей из чата через `POST .../users` (`tg_id`,
   `@username`, имя), бэк сопоставляет их с YouGile по имени (или хост указывает явно).

Назначение исполнителя при `POST /v1/tasks` идёт так: поле `assignee` (имя) сначала
ищется в **реестре** (`full_name`/`@username` → `yougile_user_id`), и только потом —
фолбэк на сопоставление с участниками YouGile по `realName`. Найденного участника
записываем и во внутренний `tasks.assignee_user_id`, и в исполнителя карточки. Если
совпадения нет — карточка создаётся без исполнителя (не ошибка).

> Чтобы человеку **реально назначилась карточка**, он должен быть **участником проекта
> в самом YouGile** (правило YouGile) — добавляет его туда хост.

## 7. Как это работает (потоки)

### 7.1 Синхронное создание задачи (`POST /v1/tasks`)

```
HTTP-запрос → валидация → service.CreateAndPublish:
  1. загрузить воркспейс (колонки) и расшифровать токен YouGile
  2. сохранить задачу в БД (approval=approved, status=todo)   ← персист ДО YouGile
  3. найти исполнителя: ListUsers → совпадение по имени
  4. создать карточку в колонке «Сделать» (+ дедлайн, +исполнитель)
  5. записать yougile_task_id в задачу
→ 201 Created с JSON задачи
```

Задача сохраняется **до** обращения к YouGile — если карточка не создалась,
задача не теряется, а ответ `502` содержит её, чтобы публикацию можно было повторить.

### Дедупликация задач

Чтобы в БД/на доске не плодились одни и те же задачи (в т.ч. сформулированные
по-разному), перед сохранением `CreateAndPublish` проверяет наличие похожих **активных**
задач тенанта (не `done`/`rejected`). Воронка:

```
новая задача
  ├─ слой 1: нормализованный матч (lower/trim)         ← дословные дубли
  ├─ слой 2: pg_trgm similarity ≥ DEDUP_SIMILARITY      ← опечатки, общие слова
  └─ слой 4 (опц., если задан OPENROUTER_API_KEY):
        LLM-судья над пулом открытых задач (cap 50)     ← «другие слова, тот же смысл»
        его вердикт ЗАМЕНЯЕТ trgm-шортлист; сбой → откат на слои 1–2
```

Слои 1–2 — детерминированные и бесплатные (в Postgres). Слой 4 — семантический,
ловит дубли, которые trgm пропускает («Починить вход» ≈ «Исправить авторизацию»),
и отсекает ложные срабатывания. (Слой 3 — эмбеддинги/pgvector — задел на масштаб,
пока не реализован.)

При находке создание **не блокируется жёстко**: API возвращает `409` со списком
кандидатов. Бот показывает их хосту; чтобы создать всё равно — повторный запрос с
`"force": true`.

```bash
POST /v1/tasks {"tenant_id":"ws-demo","title":"Починить вход"}
# → 409 {"error":"...","duplicates":[{"id":"...","title":"Исправить авторизацию",...}]}
POST /v1/tasks {"tenant_id":"ws-demo","title":"Починить вход","force":true}
# → 201 (создана принудительно)
```

### 7.2 Асинхронное событие (`POST /v1/events`)

```
HTTP-запрос {type, tenant_id, payload} → queue.Publish → 202 queued
         (асинхронно)
worker: подписан на очередь → Router.Handle → роутинг по type:
   task_create      → service.CreateAndPublish (как в 7.1)
   chat_message     → (заглушка)
   transcript_ready → (заглушка)
```

### 7.3 Смена статуса (`PATCH /v1/tasks/{id}`)

```
{status} → service.UpdateStatus:
  1. обновить статус задачи в БД
  2. если карточка опубликована: переместить в колонку статуса
  3. если status=done: дополнительно закрыть карточку (completed)
→ 200 с JSON задачи
```

## 8. Справочник API

Тело — JSON. Ошибки — конверт `{"error":"..."}`.

### `GET /healthz`
Проверка живости. Ответ: `{"status":"ok","workspaces":<n>}`.

### `POST /v1/workspaces/{tenant}/credentials`
Сохранить креды YouGile воркспейса.
**Тело:** `{"api_key":"..."}` **или** `{"login":"...","password":"...","company_name":"..."}`.
**Ответ:** `200 {"status":"stored"}`.
**Коды:** `400` нет кредов · `404` нет воркспейса · `409`/`502` YouGile отверг · `503` нет `APP_SECRET`.

### `POST /v1/workspaces/{tenant}/board/resolve`
Сопоставить колонки доски с каноническими статусами и сохранить маппинг.
**Тело:** пусто (авто-режим) **или** override `{"todo":"...","in_progress":"...","review":"...","done":"..."}`.
**Ответ:** `200 {"confident":bool,"complete":bool,"board_id":"...","mapping":{...},"assignments":[...]}`.
**Коды:** `404` нет воркспейса/доски · `409` YouGile не подключён · `502` YouGile отверг · `503` нет `APP_SECRET`.

### `POST /v1/workspaces/{tenant}/users`
Зарегистрировать участника чата (upsert по `tenant`+`tg_id`). Если `yougile_user_id`
не передан — бэк пытается сопоставить по `full_name` с участниками YouGile (best-effort).
**Тело:** `{"tg_id":"...","tg_username":"@...","full_name":"...","yougile_user_id":"..."}` (`tg_id`, `full_name` — обязательны).
**Ответ:** `200` с JSON участника. **Коды:** `400` нет `tg_id`/`full_name` · `404` нет воркспейса.

### `GET /v1/workspaces/{tenant}/users`
Список зарегистрированных участников. **Ответ:** `200 {"users":[ ... ]}`.

### `POST /v1/tasks`
Создать одобренную задачу и карточку.
**Тело:** `{"tenant_id":"...","title":"...","description":"...","assignee":"имя","deadline":"2026-06-10T18:00:00Z","force":false}`
(`description`, `assignee`, `deadline`, `force` — опционально; `deadline` в RFC3339;
`force:true` создаёт задачу в обход дедупликации).
**Ответ:** `201` с JSON задачи (включая `yougile_task_id`).
**Коды:** `400` валидация · `404` нет воркспейса · `409` **дубликаты** (тело `{"duplicates":[...]}`)
**или** YouGile не подключён · `502` задача сохранена, но карточка не создана · `503` нет `APP_SECRET`.

### `PATCH /v1/tasks/{id}`
Сменить статус + подвинуть карточку.
**Тело:** `{"status":"todo|in_progress|review|done"}`.
**Ответ:** `200` с JSON задачи.
**Коды:** `400` невалидный статус · `404` нет задачи · `409` YouGile не подключён · `502` статус сохранён, но карточка не переехала.

### `GET /v1/workspaces/{tenant}/tasks`
Список задач воркспейса (дайджест). **Ответ:** `200 {"tasks":[ ... ]}`.

### `POST /v1/events`
Принять событие в очередь (async).
**Тело:** `{"type":"task_create","tenant_id":"...","payload":{...}}`.
**Ответ:** `202 {"status":"queued"}`.
**Коды:** `400` нет `type`/`tenant_id` · `503` очередь недоступна.

**Объект задачи (ответ):**
```json
{
  "id": "uuid", "tenant_id": "ws-demo", "title": "...", "description": "...",
  "status": "todo", "approval_status": "approved", "source": "chat",
  "yougile_task_id": "uuid|null", "deadline": "RFC3339|null",
  "created_at": "RFC3339", "updated_at": "RFC3339"
}
```

## 9. Модель данных

- **`workspaces`** — тенант: один Telegram-чат ↔ один проект YouGile. `id` —
  текстовый ключ (совпадает с `workspace.yaml`), колонки доски
  (`col_todo`/`col_in_progress`/`col_review`/`col_done`), per-workspace креды
  (`yougile_login`, `yougile_api_token_enc`).
- **`users`** — участники воркспейса, маппинг на аккаунт YouGile (`yougile_user_id`).
- **`tasks`** — ядро: `approval_status` (pending/approved/rejected), `status`
  (todo/in_progress/review/done), `source` (chat/meeting), `assignee_user_id`,
  `deadline`, `yougile_task_id`, `meeting_id`, таймстемпы.
- **`meetings`** — источник задач из встреч (transcript/summary) — задел под У2/У3.

Миграции применяются автоматически на старте — идемпотентный раннер, применённые
версии хранятся в `schema_migrations`, SQL вшит в бинарь через `embed`.

## 10. Безопасность

- **Шифрование токенов:** API-ключ YouGile хранится как `BYTEA`, зашифрованный
  **AES-256-GCM**; ключ шифрования выводится из `APP_SECRET` (SHA-256). В БД нет
  ни открытого ключа, ни пароля.
- **Пароль не персистится** — используется один раз для генерации ключа.
- **Изоляция секретов:** `.env` в `.gitignore`; в логи токены/пароли не пишутся.
- **Лимиты:** тело запроса ограничено 1 МБ, строгий JSON-декод (неизвестные поля
  отклоняются), таймаут вызовов YouGile 15 с.
- **Прод:** задайте сильный уникальный `APP_SECRET` (смена секрета сделает ранее
  сохранённые токены нечитаемыми — потребуется переонбординг).

---

## 11. Надёжность и наблюдаемость

- **Структурное логирование** (`slog`, JSON): одна строка на запрос — метод, путь,
  статус, длительность; уровень растёт с классом статуса (5xx → error, 4xx → warn).
- **Recovery-middleware:** паника в хендлере → `500` со стек-трейсом в лог, сервер
  продолжает работу.
- **Ошибки YouGile** изолированы в клиенте (`APIError` со статусом и телом) и
  маппятся на понятные коды (`409`/`502`).
- **Graceful shutdown:** по SIGINT/SIGTERM HTTP завершается аккуратно, затем очередь
  дренится (в пределах буфера события не теряются).

## 13. Путь к продакшну

Точки расширения, заложенные интерфейсами:

- **Очередь:** `queue.Queue` → реализация на **NATS JetStream** (durable, ретраи,
  несколько консьюмеров) без правки воркера и продюсеров.
- **Воркер:** один goroutine → **пул** воркеров.
- **Обработчики событий:** реализовать `chat_message` / `transcript_ready`
  (вызов Claude, парсинг транскриптов, таблица `meetings`).
- **Идемпотентность:** дедупликация событий (по ключу), ретраи публикации в YouGile.
- **Хранилище:** `Repository` допускает другую СУБД без изменения бизнес-логики.
